01

The 3 Biggest AI Coding Mistakes

Observed by Andrej Karpathy (ex-Tesla AI Director, OpenAI co-founder)

Ever had this happen?

Karpathy's three pitfalls

🤫
Silent Assumptions

Faced with ambiguity, AI guesses instead of asking. You discover the wrong choice after 200 lines are written.

🏗️
Over-Engineering

You ask for a simple function, AI builds a strategy pattern + abstract class + factory. Simple task made complex.

🌊
Unrelated Changes

You ask to fix a bug, AI also renames variables, adds docstrings, changes quote styles. PR: 500 lines changed, only 2 needed.

?
Check your understanding
Which is a typical example of AI "making silent assumptions"?
Correct! "Silent assumptions" means AI picks implementation details (which fields, what format, where to save) without asking. The fix: state assumptions explicitly and ask clarifying questions first.

The Four Rules

Drop one CLAUDE.md file into your project — AI behavior improves immediately

Rule 1: Think Before Coding

When requirements are unclear, ask, don't guess. If multiple interpretations exist, list them and let the user choose.

✗ Without rules
# You: "Add a user data export feature" # AI silently decides: all fields, CSV, /tmp def export_users(): users = db.query("SELECT * FROM users") with open("/tmp/users.csv") as f: writer.writerows(users)
✓ With rules
# AI asks 4 questions first: # 1. Which users? All / active / date range? # 2. Which fields? All / anonymized? # 3. Format? CSV / JSON / Excel? # 4. How much data? Need pagination? # → Writes code after you answer

Rule 2: Simplicity First

Write the minimum code that solves the problem. Don't add "might need later" features.

✗ Without rules
# You: "Add a discount calculator" # AI builds an architecture class DiscountStrategy(ABC): def calculate(self): ... class PercentDiscount(...): ... class FixedDiscount(...): ... class DiscountFactory: ... # 30+ lines for one calculation
✓ With rules
# 3 lines. Done. def apply_discount(price, pct): return round(price * (1 - pct), 2)

Rule 3: Surgical Changes

Only touch what needs to change. Don't "improve" adjacent code.

✗ Without rules
# You: "Fix the empty email crash" # AI fixes bug AND: # - Adds username validation (not asked) # - Changes quote style (' → ") # - Adds docstring (not asked) # - Adds type hints (not asked) # PR: 47 lines changed, only 2 needed
✓ With rules
# Only 2 lines changed: null check def validate_email(email): + if not email: + return 'Email is required' # Everything else untouched

Rule 4: Goal-Driven Execution

Transform tasks into verifiable goals with checkpoints at every step.

✗ Without rules
# You: "Fix the auth system" # AI's plan: # 1. Review the code # 2. Make some improvements # 3. Test it # 4. Done # → Vague, no verification criteria
✓ With rules
# AI's plan: # 1. Write test to reproduce bug → confirm it fails # 2. Fix the code → confirm test passes # 3. Add edge case tests → confirm they pass # 4. Run full suite → confirm no regressions # → Each step has clear verification
?
Check your understanding
What does "Simplicity First" actually mean?
Correct! The key word is "current." Design patterns aren't wrong — but using a strategy pattern + factory for a single discount function is premature abstraction. Add complexity when you actually need it.

Quick Start

30 seconds, two options

Option 1: Install as Claude Code skill (recommended)

CLAUDE CODE claude /install-skill https://github.com/forrestchang/andrej-karpathy-skills

After install, Claude Code follows the 4 rules in all projects automatically.

Option 2: Copy CLAUDE.md to your project

TERMINAL curl -o CLAUDE.md https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md

Drop it in your project root. Claude Code reads it on startup. Only applies to this project.

🎉
That's it!

No configuration needed. Next time you use Claude Code, it will automatically follow the 4 rules: Think Before Coding, Simplicity First, Surgical Changes, Goal-Driven Execution.
