三大毛病，Karpathy 替你总结了

🤫
偷偷做假设

你说"导出用户数据"，它不问你要什么格式、什么范围，自己猜一个就开始写。猜错了就白写了。

🏗️
疯狂过度设计

你要一个3行就能写完的函数，它给你搞出策略模式 + 抽象类 + 配置文件，写了 30 多行。杀鸡用牛刀。

✂️
顺手乱改代码

你让它修一个 bug，它把旁边的注释改了、函数名也重构了。改的地方比你要求的多得多，还可能引入新 bug。

💡
Karpathy 的原话

"它们替你做错误假设然后一路狂奔，不管理自己的困惑，不暴露不一致性，不呈现权衡取舍。" — 原始推文 ↗

四条原则，怎么治 AI 的"毛病"

不是一个工具，不是一个框架 — 就是一个放在项目里的文本文件

它是怎么工作的？

原理非常简单：把一个叫 CLAUDE.md 的文件放在你项目根目录。Claude Code 在帮你写代码时会自动读取这个文件，按照里面的"规矩"行事。

四条原则速览

🤔
原则 1：先思考再编码

不确定就问，不要偷偷猜。有多种理解时，全部列出来让你选，而不是默默选一个就开始写。

✨
原则 2：简单优先

能 3 行解决就不要写 30 行。不为"万一以后需要"建抽象层，不加没人要求的功能。

🎯
原则 3：精准修改

只改你让改的，不碰其他代码。不"顺手"重构、不改注释、不删看着像死代码的东西。

🎯
原则 4：目标驱动执行

把模糊的任务变成可验证的目标。"修复认证"→ "写一个测试复现 bug，然后让测试通过"。

?
检查一下你的理解
你让 AI "修复邮箱为空时的崩溃 bug"。按照 Karpathy Skills 的原则，AI 应该怎么做？
正确！ 这就是第 3 条原则"精准修改"的核心。你说修 bug，就只修 bug。看到旁边有可以"改善"的代码？忍住。发现"死代码"？提醒你就好，不删。AI 做的改动越少越精确，引入新问题的概率就越低。

跟其他方案有什么不同？

市面上有不少改善 AI 编码行为的方案，它们的思路各不相同

三种不同的思路

📏
思路 A：编码哲学

Karpathy Skills 的方式 — 用几条通用原则改变 AI 的行为模式。极简，适用于任何项目。

🧰
思路 B：完整工具箱

gstack 的方式 — 提供 20+ 个专门工具（审查、发布、QA…）。功能强大但学习曲线陡。

📚
思路 C：规则集合

awesome-cursorrules 的方式 — 按技术栈收集大量规则文件。覆盖广但没有统一的"哲学"指导。

最简单的上手方式

三种安装方式，选最适合你的

方式一：一键安装（推荐）

如果你用 Claude Code，直接在终端运行：

1
安装插件

打开终端，输入下面这行命令：

TERMINAL claude plugins add forrestchang/andrej-karpathy-skills

方式二：手动下载（万能方法）

不管用什么工具，直接把文件下载到你的项目里就行：

1
下载 CLAUDE.md 到你的项目根目录

一行命令搞定：

TERMINAL # 进入你的项目目录 cd ~/你的项目/ # 下载 CLAUDE.md curl -o CLAUDE.md https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md
2
（可选）也下载示例文件

帮 AI 更好理解什么是"正确做法"：

TERMINAL curl -o EXAMPLES.md https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/EXAMPLES.md
3
开始用 Claude Code 写代码

CLAUDE.md 会被自动读取，AI 的行为会立刻改变。

🎉
就这么简单！

装完后正常用 Claude Code 写代码就行。你会发现 AI 开始主动问你问题而不是偷偷猜，写出来的代码更简洁，改动范围更精准。
