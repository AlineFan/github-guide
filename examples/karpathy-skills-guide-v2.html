01

AI 写代码时最常犯的 3 个错

Andrej Karpathy（前 Tesla AI 总监、OpenAI 联合创始人）的观察

你有没有遇到过这种情况？

Karpathy 总结的三大问题

🤫
悄悄做假设

遇到不明确的需求，AI 不问你，自己猜一个就开始写。猜错了你才发现，已经写了 200 行。

🏗️
过度设计

你只想要一个简单函数，AI 给你造了一个设计模式 + 抽象类 + 配置系统。简单需求被搞复杂了。

🌊
改一堆无关的代码

你让它修一个 bug，它顺手改了变量名、加了注释、换了引号风格。PR 里 500 行改动，实际只需要 2 行。

?
检查一下你的理解
以下哪个是 AI 写代码时"悄悄做假设"的典型表现？
正确！AI "悄悄做假设"就是在遇到不明确的需求时，不问你就自己决定了细节（比如导出哪些字段、用什么格式、存到哪里）。正确的做法是先列出假设、提出澄清问题，而不是猜。
02

四条行为准则

放进一个 CLAUDE.md 文件，AI 的行为立刻改善

准则一：先想再写

遇到不明确的需求，不要猜，要问。如果有多种理解方式，列出来让用户选。

❌ 没有准则时
# 你说："加一个导出用户数据的功能" # AI 直接开始写 —— 默认全部字段、CSV 格式 def export_users(): users = db.query("SELECT * FROM users") with open("/tmp/users.csv") as f: writer.writerows(users)
✅ 有准则后
# AI 先问 4 个问题： # 1. 导出哪些用户？全部 / 活跃 / 指定时间段？ # 2. 导出哪些字段？全部 / 脱敏？ # 3. 什么格式？CSV / JSON / Excel？ # 4. 预计多少数据？需要分页吗？ # → 等你回答后再写代码

准则二：简单优先

写能解决问题的最少代码。不要提前加"以后可能用到"的功能。

❌ 没有准则时
# 你说："加一个计算折扣的函数" # AI 造了一套设计模式 class DiscountStrategy(ABC): def calculate(self): ... class PercentDiscount(DiscountStrategy): ... class FixedDiscount(DiscountStrategy): ... class DiscountFactory: ... # 30+ 行代码，只为算一个折扣
✅ 有准则后
# 3 行搞定 def apply_discount(price, pct): return round(price * (1 - pct), 2)

准则三：精准修改

只改需要改的。不要顺手"改进"旁边的代码。

❌ 没有准则时
# 你说："修一下空邮箱崩溃的 bug" # AI 改了 bug，但同时： # - 加了用户名验证（没要求） # - 换了引号风格（单引号→双引号） # - 加了 docstring（没要求） # - 加了 type hints（没要求） # PR: 47 行改动，实际只需要 2 行
✅ 有准则后
# 只改了 2 行：加空值检查 def validate_email(email): + if not email: + return 'Email is required' # 其余代码完全不动

准则四：目标驱动

把任务转化为可验证的目标，每一步都有检查点。

❌ 没有准则时
# 你说："修一下认证系统" # AI 的计划： # 1. 检查代码 # 2. 做一些改进 # 3. 测试一下 # 4. 完成 # → 模糊、没有验证标准
✅ 有准则后
# AI 的计划： # 1. 写测试复现 bug → 确认测试失败 # 2. 修复代码 → 确认测试通过 # 3. 加边界情况测试 → 确认通过 # 4. 跑全量测试 → 确认没有回归 # → 每步都有明确的验证标准
?
检查一下你的理解
"简单优先"这条准则，核心意思是什么？
正确！关键是"当前问题"。设计模式本身没有错，但在只需要一个函数的时候就用策略模式 + 工厂模式，是过早抽象。等到真正需要的时候再加复杂性。

Before → After

用 Karpathy Skills 前后，AI 写代码的行为会发生什么变化

Before（没有准则）
你说"导出数据"，AI 猜了一堆细节直接开写
一个简单函数被写成 30 行设计模式
你让修 bug，PR 里多了 500 行"顺手改进"
AI 的计划是"检查代码 → 做改进 → 完成"
Code review 时间翻倍，因为要分辨哪些是真改动
经常需要重写，因为一开始方向就错了
After（有准则后）
AI 先问 4 个澄清问题，确认需求再动手
3 行代码搞定，等真需要再加复杂性
只改了 2 行 bug 修复，其余一行不动
每步都有验证标准："写测试 → 通过 → 无回归"
Diff 干净清晰，review 效率大幅提升
第一次就做对的概率显著提高

谁需要这个？

💻 用 Claude Code 写代码但经常需要返工的开发者 📋 受不了 AI "顺手改进"一堆无关代码的人 🎯 想让 AI 先问清楚再动手而不是猜着写的人 👥 团队想统一 AI 编码规范的技术负责人 🚀 想用最少时间获得最大改善的实用主义者
